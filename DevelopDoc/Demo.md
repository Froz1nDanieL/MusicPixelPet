# 1. 先确定项目定位

不要一开始就定义成：

> 一个能识别各种音乐类型的智能桌宠工具。

这个目标太大，也容易做偏。

更推荐定义成：

> 一个能根据系统音乐的节奏、能量、低频和播放状态做出反应的轻量桌面音乐桌宠。

也就是说，第一版核心不是“精准识别曲风”，而是：

- 音乐响了，它有反应
- 鼓点来了，它会踩点
- 慢歌时，它安静摇晃
- 快歌时，它活跃跳舞
- 暂停音乐时，它进入待机
- 全屏、学习、工作时，它不会打扰你

这个定位更容易实现，也更符合桌宠工具的体验。

# 4. 确定技术路线

既然不考虑 Python，我建议你采用：

```text
Tauri 2 + React + TypeScript + Rust
```

具体分工：

```text
Tauri：
桌面应用壳、窗口管理、托盘、打包

React + TypeScript：
桌宠界面、设置面板、动画控制、音乐状态展示

Rust：
系统音频捕获、FFT 分析、媒体信息读取、系统能力

SQLite / JSON：
保存设置、桌宠位置、动作强度、用户偏好
```

第一版动画建议不要直接上 Live2D，先用：

```text
Canvas + PNG 序列帧 / WebP 序列帧
```

原因是：

- 实现快
- 可控
- 易调试
- 方便验证动作逻辑
- 不被 Live2D 模型制作卡住

你可以等核心体验稳定后，再换成 Live2D 或 Spine。

# 7. 准备桌宠美术资源规范

桌宠项目很容易卡在美术资源上，所以你要提前定规范。

第一版建议用**帧动画**。

## 建议动作列表

第一版只准备 6 到 8 个动作：

```text
idle       待机呼吸
sleep      睡觉
nod        点头
jump       跳跃
dance      跳舞
calm       轻轻摇晃
shock      被吓到
drag       被拖拽时的表情，可选
```

不要一开始做几十个动作。

---

## 每个动作的规范

建议统一：

```text
尺寸：512 x 512 或 384 x 384
格式：PNG 序列帧 / WebP
背景：透明
帧率：24fps 或 30fps
命名：动作名_序号.png
```

例如：

```text
assets/pets/default/idle/idle_000.png
assets/pets/default/idle/idle_001.png
assets/pets/default/idle/idle_002.png

assets/pets/default/jump/jump_000.png
assets/pets/default/jump/jump_001.png
```

---

## 动作类型区分

动作最好分成两种：

### 循环动作

适合持续播放：

```text
idle
sleep
nod
dance
calm
```

### 一次性动作

播放完回到原状态：

```text
jump
shock
click
favorite
```

这样你的动画控制器会更清晰。

# 8. 设计音乐状态系统

准备阶段你就要先定义好“音乐状态”，不要边写边想。

第一版可以用这些状态：

```ts
type MusicState =
  | "silent"
  | "calm"
  | "normal"
  | "energetic"
  | "bass_heavy"
  | "shock";
```

对应关系：

| 音乐状态   | 条件             | 桌宠动作      |
| ---------- | ---------------- | ------------- |
| silent     | 没有声音 / 暂停  | idle / sleep  |
| calm       | 音量低，低频弱   | calm          |
| normal     | 普通音乐         | nod           |
| energetic  | 音量高，节奏明显 | dance         |
| bass_heavy | 低频峰值明显     | jump / bounce |
| shock      | 音量突增         | shock         |

第一版不要做太细，比如：

- jazz
- rock
- pop
- classical
- rap
- EDM

不用 AI 的情况下，直接判断曲风会不稳定。
先做“音乐状态”，效果更好。

# 9. 设计桌宠状态机

桌宠的状态不应该完全等于音乐状态。

你需要一个中间层：

```text
音乐状态 → 桌宠状态机 → 动作播放器
```

例如：

```text
silent → idle / sleep
calm → calm_sway
normal → nod
energetic → dance
beat event → 插入 jump
shock → 插入 shock
```

状态优先级建议：

```text
用户手动模式
  >
全屏 / 专注规则
  >
一次性动作
  >
音乐状态动作
  >
待机随机动作
```

这样当用户开了专注模式时，即使音乐很炸，桌宠也不会疯狂跳舞。

# 10. 设计“不打扰”规则

这是桌宠工具是否实用的关键。

准备阶段就要想清楚这些规则：

## 第一版建议做

```text
全屏应用时降低动作强度
鼠标靠近时自动降低透明度或避让
专注模式下减少动作幅度
长时间无音乐进入睡觉
用户可设置透明度
用户可设置动作强度
托盘一键隐藏
```

## 暂时不要做

```text
复杂应用识别
复杂日程联动
自动判断你是否在学习
自动生成听歌报告
```

不打扰比功能多更重要。

# 15. 准备阶段最容易犯的错误

## 错误 1：一开始就做完整播放器

不要做。
你只是音乐桌宠，不是音乐播放器。

---

## 错误 2：一开始就做精准曲风识别

不用 AI 的情况下，曲风识别容易不准。
先做“音乐状态识别”。

---

## 错误 3：一开始就上 Live2D

Live2D 很有吸引力，但会把你卡在模型、动作、授权、集成上。

第一版建议先用帧动画。

---

## 错误 4：动作太多

动作越多，不代表体验越好。
先把 6 个动作做自然。

---

## 错误 5：忽视“不打扰”

桌宠再可爱，如果挡屏幕、太吵、太跳，用户会关掉。

所以一定要准备：

```text
透明度
动作强度
专注模式
全屏弱化
一键隐藏
```
